# 功能测试指南

用于验证迁移后所有功能是否完整可用。当前项目无自动化单测，建议按下面顺序做**构建验证** + **接口健康检查** + **前端手动测试**。

---

## 一、构建与启动验证

在项目根目录执行：

```powershell
# 1. 后端构建
cd backend
npm run build

# 2. 前端构建
cd ../frontend
npm run build
```

两者均无报错即表示编译与模块引用正常。

**一键启动三端**（Windows）：

```powershell
.\start-dev.ps1
```

确认：
- 后端：http://localhost:8000 打开显示 “Smart LPR System Backend is running”
- 前端：http://localhost:5173 可打开登录/仪表盘
- AI 服务：http://localhost:8001 已启动（识别功能才可用）

---

## 二、接口健康检查（可选脚本）

在**后端与 AI 服务已启动**的前提下，可在项目根目录执行：

```powershell
.\scripts\smoke-check.ps1
```

脚本会检查：
- 后端根路径是否返回 200
- 登录接口是否返回 200 并拿到 token
- （若 AI 服务可访问）AI 健康是否正常

任一失败会以非零退出码退出，便于 CI 或本地快速回归。

---

## 三、前端功能手动测试清单

建议使用**两个账号**各测一遍：**admin**（全权限）和 **viewer**（只读），以验证权限与数据范围。

### 1. 鉴权与登录

| 步骤 | 操作 | 预期 |
|-----|------|------|
| 1.1 | 打开 http://localhost:5173，未登录 | 可看到登录入口或未认证状态 |
| 1.2 | 点击登录，输入 admin / admin123 | 登录成功，顶部显示已认证、用户名与角色 |
| 1.3 | 刷新页面 | 仍为已登录状态（token 有效） |
| 1.4 | 点击退出 | 退出成功，回到未认证状态 |

### 2. 仪表盘（dashboard）

## 四、后端 API 速查（便于用 Postman/curl 自测）

- **鉴权**：`POST /api/auth/login`（body: username, password）→ 取 `data.token`，后续请求 Header：`Authorization: Bearer <token>`  
- **当前用户**：`GET /api/auth/me`（需 token）  
- **IAM**：`GET /api/iam/users|roles|permissions`，`PUT /api/iam/users/:userId/roles` 等（需 token + iam.manage）  
- **记录**：`GET /api/plates`（可选 query: start, end, groupBy=plate），`POST /api/plates`，`GET /api/export-records`（需 token + 对应权限）  
- **统计**：`GET /api/stats/dashboard`，`GET /api/stats/daily`，`GET /api/stats/region`  
- **告警/黑名单**：`GET /api/alarms`，`GET /api/blacklist`，以及对应 POST/DELETE  
- **摄像头**：`GET /api/cameras`，`POST /api/cameras`，`PUT /api/cameras/:id`，`DELETE /api/cameras`  
- **识别**：`POST /api/recognize`（multipart，需 token + AI 服务）

## 五、测试通过标准

- **构建**：backend、frontend 的 `npm run build` 均成功。  
- **健康检查**：`.\scripts\smoke-check.ps1` 全部通过（若已配置）。  
- **手动测试**：上述清单中与当前已实现功能对应的项无阻塞性问题（白屏、500、403 与预期不符、数据错乱等）。

若某一步失败，可先查浏览器控制台、网络请求与后端日志，再结合 [项目结构](PROJECT_STRUCTURE.md)、[权限说明](IAM_RBAC.md) 和 [数据记录说明](PLATE_RECORDS.md) 排查。
